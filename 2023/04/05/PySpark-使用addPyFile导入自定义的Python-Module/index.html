PySpark:使用addPyFile导入自定义的Python Module

背景

最近在工作中遇到了一个实际的Spark使用场景：在已有一系列自定义的Python Module的情况下，如何让Spark集群中的每一个节点都能够使用这些module中的函数来处理数据？

经过调研，我们发现pyspark.SparkContext中的addPyFile能够满足这一使用场景的需要。简单来说，这个函数主要完成了两件事：

  • 将给定local或HDFS路径下的py文件拷贝至Spark集群中每一个节点下的临时目录中。其中，具体的临时路径可以通过SparkFiles.getRootDirectory()查看
  • 在每一个节点中，将这个临时路径添加到sys.path下，这意味着在每一个节点中Python解释器都可以找到这个路径下的Python Module并引入它们。

不过，上述用法在实际使用场景下遇到了一个问题。如果我们有两个不同的Python Module，它们同名却在两个不同的路径下。若想在一个Spark Job中同时使用它们，在先后对这两个路径调用addPyFile时将会遇到问题。

An error occurred while calling o2266.addFile. : org.apache.spark.SparkException: File /mnt/var/hadoop/tmp/nm-local-dir/usercache/trusted-service-user/appcache/application_1680701921935_0001/spark-c31523d7-0203-4784-aba4-6e374a019500/userFiles-601f3017-0823-4928-83ff-59eeaf55bb53/Test.py exists and does not match contents of abfss://mycontainer@mystorage.dfs.core.windows.net/Test/v2/Test.py

其原因在于，第二次调用addPyFile时，将要添加的Test.py文件无法对在临时目录下已经存在的Test.py文件进行overwrite。

方案1

起初，我们想到了如下方案来解决这个问题。

  • 将Spark的spark.files.overwrite配置设置为true
  • 在每次调用Python module前都进行importlib.reload

这样，在多次添加同名的py文件时，后一次添加总会对前一次已添加在临时目录下的同名文件进行overwrite。并且在调用module前的reload操作保证了使用的是最新添加进来的module。

但是，如果想要在同一个spark job里第二次添加path1中的Test.py文件时，输出将不会符合我们的预期。例如，下述代码将会依次输出Hello world1，Hello world2和Hello world2，而不会像我们预期的那样输出Hello world1，Hello world2和Hello world1

这是因为Spark的addPyFile方法只会处理对一个路径的第一次添加，之后对这个路径的多次添加将会被忽略（官方说法为：“A path can be added only once. Subsequent additions of the same path are ignored.”）。因此，方案1将无法覆盖到这样一个corner case。

方案2

方案1在大多数情况都能够满足场景需要，但是无法覆盖上述的corner case将决定其无法成为一个可靠的解决方案。因此，我们采用了如下的方案2来解决这一问题。

  • 首先将py文件下载到driver节点下的一个临时路径，并将其重命名为一个GUID
  • 以这个临时的local路径为参数调用addPyFile，这时这个py文件将会被分发给每一个worker节点。
  • 此时的module名为GUID，再用这个GUID来调用importlib.import_module，获得module对象，之后的调用都以其为入口

采用这种方案，每添加一个py文件，无论这个path已经被添加过多少次，无论是否曾添加过同名但不同路径的py文件，在Spark的各个节点中都会被视为不同的module。因此上述考虑到的所有corner case都可以被这个方案覆盖到，问题从而得以被解决。
